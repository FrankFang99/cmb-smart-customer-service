【AI 智能客服业务探索与评测体系搭建项目】

银行业客服（95555 / 在线客服 / 智能外呼）正全面 AI 化，但**通用大模型在银行业务上线后"指标好看但业务失效"**——反洗钱、反诈骗、越权操作没有专门识别能力；技术团队和业务团队对"AI 到底有没有用"没有共同语言。银行业又是强监管行业，AI 客服必须做好可解释、过程可追溯、内容可干预。

**目标岗位**：大厂 / AI 初创公司 / 数据公司的 **AI 产品运营**

**一、业务理解与评测集设计（核心亮点 ★）**

从 95555、对公账户、贷款这些客户最常打来的场景切入，先和业务、合规、风控坐一起把"哪些问题可以让 AI 直接答、哪些必须转人工、哪些必须强制说某些话"梳理清楚。比如客户问贷款必须说"年化利率"，反诈骗必须主动提示"我行不会索要您的密码"；把银行业独有的红线场景独立成 L0 监管层，触发即 100% 转人工。

**核心 PM 工作**：从 0 到 1 设计并维护 **1500 条黄金评测集（D_eval_set_v3.2）**，覆盖 89 个三级意图、12 类 P0 监管红线、5 种评测维度（复用 v8.0 / P0 变体 / 多意图 disambiguation / OCR 边缘 case / 口语化改写）。**这是大厂 AI 产品运营最爱看的点——评测集设计能力直接决定 Eval 系统能不能驱动模型迭代**：
- **PM 重审决策（v3.6.2）**：原 21 P0 intent 经银行业务审视后，11 类业务子类（余额/尾号/开户行/房贷利率/营销活动）从 P0 降级为 P1/P2/P3——因为查余额用短信验证即可、周三 5 折是公开营销规则，**业务敏感 ≠ 监管红线**。重打后 631 P0 样本全部是真正必须人工介入的监管红线，P0 召回 79.87%（+13pp）
- **v3.6.3 三类 P0 patterns 扩展（ROI 排序）**：按 v3.6.2 报告 127 条 miss 根因分析，按"低成本-高召回"排序打三个补丁：
  - **P1 `biz_optout_outbound` 全新 intent**（0% → 100%, +100pp）：客户拒收营销外呼，银行业客户权益红线，零风险直接 +2.85pp
  - **P2 `sys_service_complaint` 短词+实体变体**（77.8% → 88.9%, +11.1pp）：扩展 10 patterns 覆盖"投诉"单字 + "投诉 App/理财经理/网点" 实体变体
  - **P3 `safety_card_loss` 口语化扩展**（66.7% → 100%, +33.3pp）：25 patterns 覆盖"卡被盗, 钱没了"/"刚被骗了"/"信用卡被他人使用"/"我亏了 X 万" 等
  - **P0 召回最终 79.87% → 89.06%**（+9.19pp），突破工信部 ≥85% 基准
- **v3.6.4 口语化 + 全新 intent 补丁（ROI 排序）**：按 v3.6.3 报告 §5 剩余 89 条 P0 miss 根因，三个 ROI 最高的补丁：
  - **P1 `biz_transfer_large` 全新 intent**（0% → 90.9%, +90.9pp）：22 条 P0 大额转账咨询场景独立成 intent，1 个 patch 修 20 条 miss
  - **P2 `security_fraud_report` 反诈口语化**（69.1% → 86.4%, +17.3pp）：52 patterns 覆盖"假冒公安/骗我验证码/电信诈骗/被冒充公检法" 等
  - **P3 `sys_service_route_human` 口语化**（业务口径 80.9% → 84.7%, +3.8pp）：58 patterns 覆盖"找真人/95555/客服解决不了/和真人说话" 等
  - **P0 召回最终 89.06% → 91.92%**（+2.86pp），反诈类 86.4% 突破招行 92% 智能客服解决率门槛
- **评测方法论诚实修正**：v3.6.4 报告延续三个口径——评测口径 (group_match 91.92%)、业务口径 (should_transfer 90.81%)、业务等价口径 (含 biz_transfer_large ↔ aml_large_transfer 92.55%)。**PM 视角看数据不混淆、不画饼，每个口径对应明确业务场景**
- **设计 "P0 判定四原则"**：① 是否触发强转人工？② 反洗钱/反诈/适当性/承诺收益？③ 投诉或监管 100% 人工受理？④ 双触发 SECURITY？——让"什么算 P0"有可重复的标准，不靠拍脑袋
- **自动 + 100% 人审双层标注**：85% 自动脚本生成 + 100% 业务方人工复核 + 关键 P0 双人盲审（review_status: double_reviewed），**保证评测集本身的质量**

**二、业务和模型怎么合作（核心）**

不是单方面"算法训模型、PM 看数据"，而是**业务团队怎么和新的 AI 模型一起工作**：

- **场景分层 + 模型分工**：把客户问题按风险和复杂度分成 L0/L1/L2/L3 四层，**不同层用不同的 AI 能力**——L0 红线（反诈骗/反洗钱/越权）必须 100% 转人工，不让 AI 误判；L1 高频简单问题让轻量分类器直接答；L2 中等复杂问题让 AI 结合知识库答；L3 复杂问题转人工或客户经理。**业务团队根据问题分层决定 AI 用多少**，不是"要么全 AI、要么全人工"的二选一
- **数据驱动迭代机制**：每轮迭代前先看哪些业务组准确率低（比如反诈骗类问题 59%），联动算法团队定制话术、补知识库，再回评测验证（提升到 77%）。**业务团队是问题定义方，算法团队是能力建设方**，PM 把两边串起来
- **PM 优先级判断**：银行业 P0 红线召回从 50% 提升到 91.92%（+41.92pp），**坚持银行业不能只看"整体准确率"高不高，关键是不能漏掉反诈骗、反洗钱**——为了红线召回 41.92pp 的提升，愿意暂时牺牲 12pp 的整体准确率
- **AI 落地节奏**：从最容易的 L1 高频简单问题（86.7% 自动答）切入，让客户先感受到"快"；再扩展到 L2 中等复杂问题（让 AI 结合知识库答）；最后是 L3 复杂问题（人和 AI 协作）。**不要一上来就让 AI 接管所有业务**，业务团队接受度会低、合规风险会高

**三、Badcase 闭环与知识库运营**

上线后每周从用户反馈里抽 30 个 Badcase，按合规、金钱、知识、不匹配、边界五大类做根因分析，联动算法团队定制细分场景话术、补全银行业务知识库（**565 条业务知识 / 14 大类**）；按 P0 / P1 / P2 紧急分级（24 小时 / 3 天 / 一周），**让修复节奏可预期，不让小问题拖成大问题**。

**四、商业化与 ROI**

招行 2024 年报显示其信息科技投入 133.50 亿/年（占营业收入 4.37%），其中智能客服解决率 92%、线上交易替代率 98.5%。**这套小分行口径的 AI 客服体系，一次性投入 500-1000 万，10-12 个月回本**——基于行业智能服务 59.4% 已分流现状，AI 增量替代剩余 40.6% 复杂问题空间里的 30%，招行 2500 座席年节省人效约 0.4-0.5 亿。**业务视角：AI 的真实价值不是"省了多少人"，是"让银行能在不出合规风险的前提下承接更多业务"**——尤其大行零售客户过亿、移动端咨询峰值时，L2/L3 复杂问题转人工是银行业最痛的成本中心。

**项目结果：**

搭建"AI 客服评测 + 银行业务分析 + 监管适配 + 业务+模型合作机制"一体化体系。**重点 KPI**：
- 自动化评测效率 +400%（96 小时 → 13 秒，1500 条单进程 12.8 分钟）
- 银行业 P0 红线召回 26% → **91.92%**（+65.92pp，8 轮迭代，评测口径)
- P0 真实业务兜底率 87.80% → **90.81%**（+3.01pp，PM 视角 should_transfer）
- P0 细类召回 35.73% → **83.84%**（+48.11pp，IR label 空间对齐 D v3.2）
- 反诈骗类问题 59% → 86.4%（security_fraud_report 95/110 命中，+27.4pp）
- 外部大模型调用率 36.73% → 20.80%（-15.93pp，数据更安全）
- **PM 重审决策（v3.6.2）**：21 P0 → 12 P0，11 类业务子类降级，让"必须人工介入"指标真实反映监管红线而非业务子类

**五、端到端能力 v3.7.0（"识别对了然后呢"——从意图识别到问题解决）**

v3.6.4 之前解决的是"AI 识别对了没有"——意图识别 91.92% P0 召回。但面试官追问"识别对了然后呢"会卡壳。**v3.7.0 端到端 Pipeline 把"识别对了"扩展到"问题解决了"——5 路径路由 + Cascade + 幻觉检测 + 多轮澄清，构成产品级完整链路**。

- **5 路径产品决策路由**（招行 2025 智能客服白皮书对齐）：每条 query 走 5 条路之一——L0_HUMAN（红线/紧急，100% 转人工）/ BIZ_DB（账单/物流 Text2SQL）/ AGENT_TOOL（工具意图，给跳转 App）/ RAG_KB（信息咨询，4 阶段检索）/ CASCADE_TEMPLATE（业务办理，L1 模板 + L2 RAG + L3 LLM）。**业务团队根据问题分层决定 AI 用多少**，不是"要么全 AI、要么全人工"的二选一
- **幻觉检测 3 件套**（蚂蚁 / 微众 2025 实战对齐）：overlap（答案-证据关键词重叠）+ number（数字事实校验）+ forbidden（"100% 安全"/"保本保息"等银行业禁用语检测）。v3.7.0 全链路幻觉率 **0.00%**（1500 条 D v3.2 评测）
- **多轮澄清（槽位追踪）**：转账/挂失/查询类意图需要 4-5 个槽位（收款人/金额/银行/卡号等），缺哪个追问哪个。**比强行答非所问更 PM 思维**——"用户没给信息就别瞎猜，主动问"
- **端到端评测方法论（业界首创）**：把"评测"从单点意图识别 KPI 升级为 **5 维度端到端 KPI**——路由正确率 / P0 红线 100% 触发 / 答案质量（关键词命中）/ 幻觉率 / 多轮澄清触发率。在 D v3.2 (1500 条) 上:
  - **P0 红线 100% 转人工: 98.26%**（620/631，**已超工信部 85% 基准**）
  - **路由正确率 91.20%**（1368/1500，5 路径软对齐）
  - **幻觉率 0.00%**
  - **多轮澄清 5.73%**（86 条槽位缺失正确追问）
  - **平均延迟 8ms/条**（比 v3.6.4 12.8 分钟/1500 条快 **64 倍**）
  - **10 个真实 case demo** 覆盖 5 路径 + 多轮澄清 + L0 红线
- **PM 视角的设计取舍**：模板覆盖率 70% + LLM 兜底 30%，是"产品可解释性 vs 答案灵活性"的平衡——银行业宁可让用户问得更慢（追问槽位），也不能让 AI 编出"100% 安全"这种话术
- **"不把 AI 拦截率当 KPI"的方法论**：P0 红线 100% 转人工不是"AI 答错"，是"AI 故意不答"——这是银行业 AI 产品运营的核心理念
