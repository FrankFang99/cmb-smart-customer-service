## 招商银行佛山分行"小招"AI 智能客服 — 业务探索与多维评测体系搭建 (v3.12.1)

**项目背景**: 招商银行佛山分行 (360 万零售客户) App 内置 AI 客服"小招"是客户体验关键一环。分行原规则化智能客服 (关键词 + 预置问答) 能解决 30-40% 的标准化问题, 但零售客户问法多样、覆盖广、行业红线复杂 —— AI 直接接入客服后普遍"指标好看但业务失效": 复杂业务规则没人专门识别、业务与技术团队对"AI 到底有没有用"没共同语言, 强监管行业又要求可解释可追溯可干预。**这个项目的核心命题: 业务团队怎么和新的 AI 模型一起工作**。本项目从 0 到 1 把分行规则化客服升级为 AI 智能客服, v3.12.0 完成多维评测体系闭环。**行业对标 (2025-08 中银协)**: 银行业客服中心 2024 年智能技术综合使用率 93%, 大模型探索渗透率 31%, 人工电话平均接通率 94.18%, 人工电话客户满意度 99.46% 连续 4 年高于 99%。

**核心工作**:

1. **业务优先级判断与分层策略**: 基于 10+ 专家访谈与分行历史问答日志调研, PM 决定"优先做标准化高频低风险问题、暂缓个性化高风险问题" —— 账单/网点/活动类占客服量 67% 且风险可控优先做, 投诉/理财类个性化强且风险高暂缓。将意图拆解为 7 大类 84 子类 (信息查询 / 业务办理 / 咨询 / 活动 / 安全 / 用户反馈 / 系统), 针对不同类型制定差异化策略。**PM 重审把 21 类红线降级到 12 类** —— 业务敏感 ≠ 监管红线, 避免把业务子类误标为红线。**v3.12.0 业务权重表**: 引入 P0=10x (红线 100 万一次) / P1=5x (业务理财账单) / P2=2x (长流程) / P3=1x (闲聊), 加权准确率比平权准确率 +2.02pp (90.24% → 92.26%), 业务权重视角比平权更有意义。

2. **RAG 知识库完善与三方协作**: 补全监管类文件 / 业务专家 Q&A 对 / 产品手册 / 专业术语四类源, 建立检索-生成-端到端三层指标定位 RAG 失败模式; 与分行业务、合规、IT 协商制定维护流程与分工并全程留痕。**P0 红线类问题设计成"100% 转人工 + AI 不答业务"**, 避免大模型编造反诈话术。**行业对标 (招行 2018 科大讯飞方案)**: 招行 95555 智能语音导航已覆盖 27 大类 450+ 业务场景, 证明银行业 AI 客服"业务规则层级化"是必要设计。

3. **意图识别三层架构与红线召回**: 维护关键词规则-小模型-大模型三层意图识别架构 (v3.6.1 → v3.10.1 → v3.11.0 → v3.12.0 → v3.12.1 九轮迭代), 复用关键词规则资产, **针对合规风险高的意图补充训练样本不抄权重**。**v3.12.1 实测 (D v3.2 dedup 1076 条)**: P0 召回 97.91% (422/431, -2.09pp 是 9 条难样本) / P1 准确 **74.70% (248/332, +3.62pp)** / P2 95.48% / P3 100% / 5 轮 bootstrap 稳定性 std=0.93pp / **P0 不漏检 std=0.50pp / worst-case 97.20%**。**v3.12.1 对抗性评测 (100 条 6 类)**: 整体 **95% (+56pp)** 识别率, **Prompt Injection 100% / 钓鱼话术 100% / 越权诱导 100% / 越界 100%** ✓, 反向诱导 90% / 情绪对抗 85% (剩余 5 个 miss 是真实威胁/自杀危机,应走危机干预 patches v3.13.0)。**宁愿暂时牺牲 2pp P0 准确率也要保住 56pp 对抗性拦截** (银行业生产环境必须 ≥85% 拦截结构性攻击)。**行业对标 (2025 IDC)**: 银行业 AI 客服标准化问题一次解决率 85-95%, 意图识别准确率行业标杆 92-95%, 本项目 P0 红线召回远超行业 (行业规则化无 P0 概念),对抗性 95% 拦截达生产可用。

4. **多维评测体系搭建 (v3.12.0 升级)**: 从 0 到 1 设计 1500 条黄金评测集覆盖 86 三级意图 + 12 类红线 + 5 种场景; v3.12.0 升级为多维闭环: **业务权重准确率** (P0=10x) + **多轮对话评测** (12 个真实业务对话流 66.7% 通过) + **稳定性指标** (5 轮 bootstrap std=0.72pp) + **业务 KPI** (FCR / CSAT / NPS / 端到端延迟) + **对抗性评测** (100 条 6 类)。整套方法论覆盖"扩样本 → 跑基线 → 写 patch → 双验证 → 沉淀负例 → 业务权重 → 多轮 → 稳定性 → 业务 KPI → 对抗性 → 下一轮"完整 Loop Engineering 闭环。

**PM 视角的核心反思** (项目 5 大不足 + v3.12.0 解决情况):
1. **没有真实 95555 用户日志** — 项目期受分行合规审批 + 数据脱敏流程限制, **0 条真实 query**; 招行 95555 日均 80 万通, 理论可采 ~2 亿条真实 query。**v3.12.0 仍 0 条, 上分行后第一件事是用真实数据跑回归**。
2. **评测样本都是单轮对话** — v3.12.0 已升级 12 个真实多轮业务对话流 (招行 95555 实战场景), 通过率 66.7%。
3. **未标注业务权重** — v3.12.0 已实现 P0=10x / P1=5x / P2=2x / P3=1x 业务权重表, 加权准确率 92.26% (+2.02pp)。
4. **只考虑正确率, 没考虑稳定性** — v3.12.0 已实现 5 轮 bootstrap 重采样, 平权 std=0.72pp, P0 漏检 std=0.00pp。
5. **对抗性 / 越界 / 钓鱼样本未系统化** — v3.12.0 已实现 100 条 6 类对抗性评测集, 暴露真实漏洞 (Prompt 0% / 钓鱼 0% / 越权 11% / 越界 10%)。

**评测集设计的 PM 取舍**: 评测集 P0 占比 40% (生产实际 < 5%) 是 PM 主动设计的压力测试集, 目的是把红线召回压到极限 —— 银行业 AI 客服核心 KPI 是"P0 红线召回"而不是"整体准确率", 压力测试集比均匀分布更能反映系统稳定性。**49.81% 转人工率 / 50.19% FCR 是评测集口径, 不是生产口径**; 按生产 P0 占比重算, FCR 应能到 88-92% (跟招行 2025 智能客服白皮书 92% 一致)。

**拒答率 0% 是核心设计取舍**: 任何 query 必须落到 answer 或 transfer_human 两个动作之一, **要么给答案、要么直接转人工, 没有"不知道"中间态** —— 避免了用户被反复踢皮球的体验。行业普遍拒答率 5-15%, 本项目 0% 兜底。
