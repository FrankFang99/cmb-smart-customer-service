PM 视角：转人工占比 49.81% 怎么解读？

评测集设计的 PM 取舍：49.81% 转人工占比 ≈ 100% P0 转人工 + 29% P1 误伤 + 6% P2 转人工。这是PM 主动设计的压力测试结果——D v3.2 评测集 P0 占比 40%（生产实际 < 5%），目的是把红线召回压到极限。银行业 AI 客服核心 KPI 是 P0 红线召回，而不是整体准确率，压力测试集比均匀分布更能反映系统稳定性。

生产口径锚点（招行 2025 智能客服白皮书）：95555 日均 80 万通，智能客服解决率 92%，生产实际转人工率 ~8%。本项目漏斗若按生产 P0 占比重新跑，AI 直接回答率应能到 88-92%。

拒答率 0% 是核心设计取舍：传统客服系统常出现"这个我答不了 / 请转人工 / 未知错误"这类黑屏体验。本项目设计取舍是任何 query 必须落到 answer 或 transfer_human，要么给答案、要么直接转人工，没有"不知道"中间态。行业普遍拒答率 5-15%，本项目 0% 兜底。

本项目最大遗憾：所有数字基于 1076 条黄金评测集 + 206 条新构造样本，0 条真实 95555 用户日志 query。招行 95555 日均 80 万通，本项目期 8 个月理论可采集 ~2 亿条真实 query。真实意图分布 / NPS / 满意度 / 解决率待分行上线后用真实数据补做——这是 PM 视角下个迭代的核心补做项。

简历级 PM 视角总结 (v3.12.0)

v3.12.0 实测

核心命题

银行业 AI 智能客服不是技术问题，是业务问题。招行 2025 NPS 74.01、满意度 99.46% 是行业基线，AI 落地最大的瓶颈是数据孤岛和跨部门协同，不是模型本身。

合规 > 一次解决率

P0 红线 100% 转人工，主动牺牲 5-10pp 一次解决率换合规。v3.12.0 实测：P0 转人工率 100% (431/431)，P0 漏检 std=0.00pp（5 轮 bootstrap 绝对稳定）。

v3.12.0 业务权重视角 (P0=10x 加权)

业务权重准确率 92.26% (5940/6438)，比平权 90.24% 高 +2.02pp，符合银行业"红线优先于业务"价值观。P0=10x（100 万一次的代价）/ P1=5x（业务）/ P2=2x（长流程）/ P3=1x（闲聊）。

v3.12.0 多维评测体系闭环

业务 KPI：CSAT 4.33/5（优秀）/ NPS 75（远超金融 30 基准）/ FCR 50.19%（评测集口径）/ 端到端 P50=98ms / P95=11s / P99=13.7s（含 LLM 兜底）。

多轮对话：12 真实业务对话流 66.7% 通过率 (8/12)，对话完成 100% / 槽位填充 100% / 合规风控 83.33% / 上下文保持 83.33%。L2 多轮 0% 通过待修。

稳定性：5 轮 bootstrap 平权 std=0.72pp，P0 漏检 std=0.00pp（绝对稳定）。

对抗性：100 条 6 类对抗性 query 整体 39% 识别率，反向诱导 95% / 情绪对抗 85% ✓，但 Prompt Injection 0% / 钓鱼话术 0% / 越权诱导 11% / 越界 10% ⚠️ 暴露真实结构性攻击漏洞。

项目 5 大不足 v3.12.0 解决情况

1. 没有真实 95555 用户日志 — ❌ 仍 0 条 (项目期合规审批限制)，上分行后第一件事拉真实数据回归。

2. 评测样本都是单轮对话 — ✅ v3.12.0 已补做，12 真实业务多轮对话流 66.7% 通过率。

3. 未标注业务权重 — ✅ v3.12.0 已补做，P0=10x/P1=5x/P2=2x/P3=1x，加权准确率 92.26%。

4. 只考虑正确率，没考虑稳定性 — ✅ v3.12.0 已补做，5 轮 bootstrap std=0.72pp，P0 漏检 std=0.00pp。

5. 对抗性 / 越界 / 钓鱼样本未系统化 — ✅ v3.12.0 已补做，100 条 6 类对抗性评测集，暴露真实漏洞。
