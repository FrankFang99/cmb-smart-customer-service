### 1.4 FCR/转人工率口径说明 (PM 视角)

**50.19% FCR / 49.81% 转人工率是评测集口径, 不是生产口径。**

- D v3.2 评测集 P0 占比 40% (PM 主动设计的压力测试集), 生产 P0 实际 < 5%
- 按生产 P0 占比重新算, FCR 应能到 88-92% (跟招行 2025 智能客服白皮书 92% 一致)
- **49.81% 转人工率 = 100% P0 + 29% P1 + 6% P2**, 都是 PM 主动设计的结果
- **核心价值**: 0% 拒答率 (任何 query 必须落到 answer 或 transfer_human), 行业普遍 5-15% 拒答意味着 5-15% 用户被"我不知道"踢皮球

### 2.2 稳定性指标 (5 轮 bootstrap)

| 指标 | Mean | Std | Min (worst-case) | Max | Range |
|---|---|---|---|---|---|
| **平权准确率 (5 轮)** | 90.95% | **0.72pp** | 90.24% | 92.01% | 1.77pp |
| **P0 不漏检 (5 轮)** | **100.00%** | **0.00pp** | **100.00%** | 100.00% | 0.00pp |

**核心洞察**:
- **平权准确率 std=0.72pp, 5 轮波动 < 2pp** — 生产稳定性达标
- **P0 不漏检 100% 绝对稳定 (std=0.00pp)** — 银行业红线核心 KPI 稳如磐石

**方法学说明**: 5 轮 bootstrap 重采样 (seed=42) 对 1076 query 做有放回重采样, 算平权准确率 + P0 不漏检的 std/min/worst-case。**比"一次跑通结果"更接近生产稳定性真实表现**。

## 3. v3.12.0 关键方法论贡献

### 3.1 业务/模型目标 KPI 双视角

v3.11.0 之前简历 KPI 全部是**模型视角** (准确率/召回率/延迟), **业务视角 KPI 严重缺失**。v3.12.0 重设计:

| 视角 | v3.12.0 实现 | PM 价值 |
|---|---|---|
| **业务目标** | 业务权重准确率 / FCR / CSAT / NPS / 转人工率 | 业务能直接 ROI 算钱 |
| **模型目标** | 模型准确率 / 对话轮次 / 端到端延迟 | 算法/工程看的 |

### 3.2 Loop Engineering 闭环升级

原闭环: **扩样本 → 跑基线 → 写 patch → 双验证 → 沉淀负例 → 下一轮**

v3.12.0 新增环节:
- **业务权重标注** — 每个 query 标业务权重, 加权准确率比平权准确率更有业务意义
- **多轮评测** — 12 个真实业务对话流, 不只是单 query 准确率
- **稳定性指标** — 5 轮 bootstrap, std/min/worst-case, 不只"一次跑通"
- **业务 KPI** — FCR/CSAT/NPS, 不只技术指标
- **对抗性评测** — 100 条结构性攻击 query, 不只"标准 query"

### 3.3 PM 视角的核心反思

| 维度 | v3.11.0 之前 | v3.12.0 改进 |
|---|---|---|
| 模型 KPI 完整度 | 高 (准确率/召回/延迟) | 持平 |
| 业务 KPI 完整度 | **低** (几乎为零) | **高** (业务权重/FCR/CSAT/NPS) |
| 多轮能力评测 | 无 | 12 场景 66.7% 通过率 |
| 稳定性评测 | 无 (单次跑通) | 5 轮 bootstrap, std=0.72pp |
| 对抗性评测 | 无 | 100 条 6 类, 暴露真实漏洞 |
| 真实用户日志 | 0 条 (项目期未接入) | 仍 0 条 (上分行后补做) |
