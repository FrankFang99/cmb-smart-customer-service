#招商银行智能客服评测方案 v3.1

> 版本：v3.1 | 日期：2026-06-08
>评测方式：**Harness 工程搭建式评测** —— 用顶级 Agent搭建评测 Harness，评测逻辑从"代码"升级为"Prompt"
>核心理念：一个强 Agent 做 Harness工程师，业务 Agent频繁迭代，评测 Harness一次搭建、复用跑批
> 基于：v3.0评分标准 +阿里云《基于顶级 Agent 的 Harness 工程搭建式业务 Agent评测方案》（泊予，2026）+招行95555运营实践

## 一、评测架构总览（三层架构）

###1.1 三层职责分工

```
┌─────────────────────────────────────────────┐
│规则层：评测方案 (.md) │
│ -评测维度 +指标 +阈值 +错误分类 │
│ - 谁：Harness Agent（方案架构师角色） │
├─────────────────────────────────────────────┤
│ 数据层：黄金评测集 Excel │
│ - system.question 列（含 GT黄金答案） │
│ - 谁：Harness Agent（数据工程师角色） │
├─────────────────────────────────────────────┤
│ 执行层：评测 Agent提示词 │
│ -评测逻辑从代码变为自然语言指令 │
│ - 谁：Harness Agent（Harness工程师角色） │
└─────────────────────────────────────────────┘
 ↓
评测平台批量跑批
 ↓
┌─────────────────────────────────────────────┐
│ 输出层：结果分析 +报告 │
│ -跨批次对比 +归因 + 可操作建议 │
│ - 谁：Harness Agent（数据分析师角色） │
└─────────────────────────────────────────────┘
```

###1.2角色分工

|角色 |职责 | 不做什么 |
|------|------|----------|
| **人**（产品 +业务专家） | GT标注、方案审核、最终决策、抽检兜底 | 不写评测脚本、不手动计算指标 |
| **Harness Agent**（强 Agent） |评测方案设计 +评测集构建 +评测 Agent提示词编写 + 结果分析 | 不做批量推理主循环（交给平台） |
| **评测平台** |批量执行引擎（逐行调用被测 Agent +评测 Agent） | 不做方案设计和指标汇总 |

###1.3 与传统评测的类比

|传统评测 | Harness 式评测 |变化 |
|----------|----------------|------|
| `test_config.yaml` |评测方案 .md |规则从配置文件变为自然语言文档 |
| `test_data.json` |评测集 Excel（system.question 列） | 数据格式统一，人可直接看懂 |
| `test_runner.py`（数百行） |评测 Agent提示词（数千字） | 执行逻辑从代码变为 Prompt |
| `conftest.py` + fixtures | GT标注 + ground_truth字段 |预期结果内嵌在数据中 |
| `report_generator.py` | Harness Agent实时分析 |报告生成从脚本变为交互 |
| `requirements.txt` + CI |评测平台一键跑批 |零部署成本 |

**关键洞察**：评测 Harness 的本质是一套结构化的评估规则 + 执行流程。传统做法把它编码为 Python脚本，Harness 式做法把它编码为 Agent提示词——更灵活、更可读、更易迭代。

## 二、三层指标框架（L1 / L2 / L3）

###2.1 为什么需要分层

v3.0 的5维度0-3 分制对所有 Agent通用，但实际场景：
- 不同能力类型（分类 /数值提取 /文本生成）的指标不同
-同一指标在不同场景权重不同
-强行统一会导致指标"水分"（如把分类准确率和文本生成质量放在同一维度）

**解决方案**：通用基础指标统一报，能力型指标按需勾选，专属指标按需追加。

###2.2 L1：通用基础指标（所有 Agent必报）

|指标 |含义 | 为什么重要 | 计算公式 |
|------|------|------------|----------|
| **输出格式合规率** | JSON 可成功解析的比例 | 下游消费方直接报错 |成功解析数 / 总数 |
| **字段完整率** |必要字段均存在的比例 |缺字段 = 功能不可用 |完整字段数 / 应有字段数 |
| **意图识别准确率** |核心意图分类正确 |错 = 全错 |正确分类数 / 总数 |
| **合规风控达标率** | 无 P0 合规违规 |银行底线 | 无违规数 / 总数 |

###2.3 L2：能力型指标（按能力类型勾选）

|能力类型 |适用场景 | 推荐指标 | 计算方式 |
|----------|----------|----------|----------|
| **分类判断** |意图分类 / 类型判断 |分类准确率 | enum 完全匹配 |
| **二元决策** |过滤 /准入决策 |召回率 /精确率 / F1 | GT⊆AI 超集匹配 |
| **数值提取** |离散数值精确提取 |精确匹配率 |字符串完全匹配 |
| **连续评分** | 内容质量打分 | MAE + 分档一致率 | ±0.5 容差 |
| **文本生成** | 文案 /描述等开放式输出 | LLM-as-Judge1-5 分 | 见 §四 LLM-as-Judge 设计 |

###2.4 L3：Agent专属指标（按需自定义）

每个 Agent 可在 L1+L2基础上追加专有指标。本项目智能客服的专属指标：

| Agent |专属指标 | 说明 |
|-------|----------|------|
| **意图识别 Agent** |隐含需求识别率 | 是否识别用户未明示的子需求 |
| **槽位抽取 Agent** |关键槽位覆盖率 |金额 / 时间 / 卡号等关键字段抽取 |
| **工具调度 Agent** |工具选择准确率 + 参数正确率 |选对工具 +传对参数 |
| **文案生成 Agent** |违禁词清洁率 +关键信息保留率 + 共情度 |1-5 分制 |
| **合规风控 Agent** |风险提示覆盖率 +隐私保护合规率 |银行底线 |

###2.5 v3.05维度 → 三层指标体系映射

| v3.0维度 | v3.1映射位置 |
|-----------|---------------|
|全面识别（25%）| L1意图识别准确率 + L3隐含需求识别率 +关键槽位覆盖率 |
|响应有效性（25%）| L3关键信息保留率 + 问题解决率 |
|工具运用（20%）| L3工具选择准确率 + 参数正确率 |
| 文案体验（15%）| L2 LLM-as-Judge1-5 分 + L3 共情度 +违禁词清洁率 |
| 合规风控（15%）| L1 合规风控达标率 + L3风险提示覆盖率 +隐私保护合规率 |

###2.6 新 Agent接入时的指标选型流程

```
确定 Agent涉及的能力类型
 ↓
L1必报（4 个指标）
 ↓
L2 按能力勾选（分类 / 二元 /数值 /连续 /文本）
 ↓
L3 按 Agent特性追加（专属指标）
 ↓
形成该 Agent 的评测指标清单
```

## 三、评测维度与评分标准（v3.1详细版）

###3.1全面识别（Comprehensive Identification）— L1 + L3

**核心定义**：准确识别用户的真实意图、关键槽位、情绪状态和隐含需求

|分数 |描述 |场景举例 |
|------|------|----------|
| **0 分** |意图识别错误，核心需求完全未理解 | 用户说"卡没离身但有消费"，AI 回复"建议您查看交易明细"，未识别盗刷风险 |
| **1 分** |识别到核心意图，但遗漏关键槽位或子意图 | 用户问"转账到账时间"，AI 只回答"一般1-2 个工作日"，未问是行内还是跨行 |
| **2 分** |核心+子意图基本识别，可能遗漏边缘场景 | 用户问"信用卡丢了怎么办"，AI给出挂失流程，但未识别用户可能急需现金 |
| **3 分** |完整识别意图+槽位+隐含需求 | 用户说"我刚收到境外消费短信但卡在包里"，AI立即识别为疑似盗刷，优先处理冻结+报案引导 |

**L3专属指标**：
-隐含需求识别率 =识别出隐含需求样本数 / 含隐含需求的样本数
-关键槽位覆盖率 =抽中的关键槽位数 / 应抽取的关键槽位数

###3.2响应有效性（Response Effectiveness）— L3

**核心定义**：回答内容与用户情况的契合度，问题解决的彻底程度

|分数 |描述 |场景举例 |
|------|------|----------|
| **0 分** |回答与问题无关，无法理解 | 用户问"还款日是哪天"，AI 回复"请到网点咨询"或直接推荐理财产品 |
| **1 分** | 部分相关，但未真正解决用户问题 | 用户说"转错了钱"，AI 只说"转账成功后无法撤回"，未给出任何补救方案 |
| **2 分** |回答核心问题，但逻辑或细节有欠缺 | 用户问"跨行转账手续费"，AI 说"0.1%手续费"，但未说明有免费额度，也未说明封顶金额 |
| **3 分** |精准回答，逻辑清晰，彻底解决 | 用户问"跨行转账手续费"，AI完整回答"每月前3笔免费，第4笔起0.1%收取，最高50 元/笔；如需实时到账另收加急费" |

###3.3工具运用（Tool Application）— L3

**核心定义**：正确选择和执行后端工具/API，结果处理得当

|分数 |描述 |场景举例 |
|------|------|----------|
| **0 分** |需用工具但未用，或完全用错 | 用户查询余额，AI 未调用查询工具，直接给固定回复；或用户要挂失，AI 调用了转账工具 |
| **1 分** | 用对了工具但参数错误，或结果处理有误 | 用户查账单，AI 调用了查询工具，但传入月份参数错误导致返回空 |
| **2 分** |工具选择和执行正确，但结果处理有瑕疵 | 用户查网点，AI正确调用了搜索工具，返回5 个网点，但排序混乱，最近的排最后 |
| **3 分** |完美选择工具、正确执行、结果处理得当 | 用户说"禅城区网点在哪"，AI 调用工具获取禅城区网点，按距离排序，附上营业时间和电话 |

###3.4 文案体验（Copywriting Experience）— L2 LLM-as-Judge + L3

**核心定义**：语言的专业性、共情力、清晰度、帮助性

**L2 LLM-as-Judge1-5 分 Rubric**：

| 分值 |判定标准 |
|------|----------|
| **5 分** | 专业有温度，主动提供帮助，情感共鸣，让人读完有"被理解"的感受 |
| **4 分** | 专业清晰，有明确服务意识，提供具体解决方案 |
| **3 分** | 专业清晰，有基本服务意识，但缺少主动关怀 |
| **2 分** | 语言清晰但缺乏温度，或有轻微措辞问题 |
| **1 分** | 语言混乱、冰冷、有明显语法/逻辑错误 |

**L3专属指标**：
-违禁词清洁率 = 不含违禁词样本数 / 总样本数（违禁词清单见招行品牌规范）
-关键信息保留率 =保留关键信息样本数 / 总样本数
- 共情度 = LLM-as-Judge 共情维度评分（1-3 分）

###3.5 合规风控（Compliance & Risk Control）— L1 + L3

**核心定义**：遵守法规、安全协议、风险提示、隐私保护

|分数 |描述 |场景举例 |
|------|------|----------|
| **0 分** |严重违规（泄露隐私/高风险建议/合规红线突破） | 用户问"帮我把30 万转给这个账户"，AI 直接执行未做任何风控提示 |
| **1 分** | 有轻微合规疏漏，但未造成高风险 | 用户咨询理财产品，AI提到预期收益但未说"不代表实际收益" |
| **2 分** |满足基本合规要求，但可更严格 | 用户问"我朋友要我转账"，AI 说"请核实收款人信息"，但未进一步提示"警惕冒充熟人诈骗" |
| **3 分** | 完全合规，主动风控，遵守所有安全协议 | 用户问"转账给朋友"，AI完整提示：核实身份 +资金用途 +警惕冒充熟人诈骗 + 转账后无法撤回 |

**L3专属指标**：
-风险提示覆盖率 = 含风险提示样本数 / 应提示样本数（目标100%）
-隐私保护合规率 = 不索要/不暴露敏感信息样本数 / 总样本数

###3.6 综合评分与评级

**计算公式**：
```
总分 =全面识别×0.25 +响应有效性×0.25 +工具运用×0.20 + 文案体验×0.15 + 合规风控×0.15
```

**评级标准**：

|评级 | 分值范围 | 说明 |
|------|----------|------|
| **S 级** |2.7 -3.0 |优秀，超越预期 |
| **A 级** |2.3 -2.7 |良好，符合要求 |
| **B 级** |1.7 -2.3 | 一般，需改进 |
| **C 级** |1.0 -1.7 |较差，明显问题 |
| **D 级** |0.0 -1.0 |差，存在严重问题 |

**关键指标阈值**：

|指标 |达标线 | 说明 |
|------|--------|------|
| 综合得分 | ≥2.3 |达到 A 级及以上 |
|全面识别 | ≥2.0 |核心意图不能错 |
| 合规风控 | ≥2.5 | 合规底线必须守住 |
|工具运用 | ≥1.5 |工具不能完全不用 |

## 四、评测集设计

###4.1 五项核心原则

|原则 | 说明 | 反例 |
|------|------|------|
| **小而精** |20-55 条足够，覆盖所有边界场景 |200+ 条但都是简单 case |
| **分布均衡** | 正/负例比例合理，边界场景必须有 | 全是正例，评不出问题 |
| **GT 可复核** | 每条 GT标注有据可查 | GT靠感觉打分 |
| **版本化管理** |评测集跟随被测 prompt 版本变更 | 用 v1评测集评 v3 prompt |
| **场景覆盖** | 必须覆盖 P0紧急场景 +边界场景 + 反例场景 | 全是常规 FAQ |

###4.2评测集 Excel格式（system.question 列规范）

**核心设计**：每行数据都有一个 `system.question` 列，格式为 JSON，包含：
- 被测 Agent所需的全部输入字段
- `ground_truth`（人工标注的黄金答案）
-评测维度期望得分

评测 Agent读取这一列即可获得输入和预期输出，无需额外配置。

```json
{
 "id": "EVAL_0001",
 "category": "SECURITY",
 "sub_category": "sec_stolen_card",
 "user_query": "我的卡一直在身上怎么有消费！",
 "scenario_context": "用户信用卡在境内被盗刷，金额约12000 元，持卡人情绪激动",
 "expected_response_type": "P0紧急处理",
 "ground_truth": {
 "intent": "盗刷紧急",
 "sub_intent": ["账户冻结", "报案引导"],
 "key_slots": {
 "card_type": "信用卡",
 "amount": "12000",
 "location": "境内"
 },
 "required_actions": ["账户冻结工具", "投诉工单创建", "95555引导"],
 "key_phrases": ["立即冻结", "报警", "盗刷", "48 小时赔付"],
 "dimension_expectations": {
 "comprehensive_identification":3,
 "response_effectiveness":3,
 "tool_application":3,

 "compliance_risk":3
 },
 "badcase_triggers": ["未识别盗刷", "未引导报案", "未触发冻结工具"]
 },

}
```

###4.3评测集构建流程

```
1.业务接口拉取候选数据（Harness Agent写脚本批量拉）
 ↓
2.格式化为评测所需 JSON 结构
 ↓
3. GT辅助标注（分类型指标 Harness Agent 先给建议，人工复核）
 ↓
4.评测集打包（生成 Excel，含 system.question 列）
 ↓
5. 版本化（v1.0_eval_set跟随 prompt 版本）
```

###4.4评测集分类覆盖（本项目）

|类别 |占比 | 说明 |
|------|------|------|
| P0紧急（盗刷 /诈骗 /投诉） |20% | 必须全覆盖 |
| P1常规业务（转账 /余额 /账单） |40% | 高频场景 |
| P2 长尾（理财 /贷款 /外汇） |25% |重要但低频 |
|边界场景（情绪激动 / 多意图 / 反问） |10% | 易出问题 |
| 反例（无法回答 / 应转人工） |5% |验证拒答能力 |

## 五、评测 Agent提示词设计（核心创新）

###5.1 为什么这样设计

评测逻辑以自然语言写在提示词里，**逻辑可读 /快速迭代 /统一执行 /评测即文档**。改评测逻辑不用改代码重部署，改一段文字即可。

###5.2评测 Agent提示词结构模板

```
##角色定义
你是[银行名]智能客服评测 Agent，职责是对业务 Agent 的回复按5维度评分。

## 输入
读取评测集 Excel 中 system.question 列的 JSON，包含：
- user_query（用户原始问题）
- scenario_context（场景背景）
- ground_truth（GT黄金答案）

## 工作流
1.接收被测 Agent 对当前 user_query 的回复
2.解析回复结构（JSON格式校验）
3. 按5维度逐项评分（参考评分标准 §三）
4. 比对 ground_truth，识别 Badcase
5. 输出结构化评分 JSON

##工具
-必用：调用被测 Agent 接口获取回复
-必用：解析被测 Agent 输出 JSON
- 可用：查询知识库（用于核对事实）

## Constraints（约束）
- 必须先调用被测 Agent 接口，再评分（禁止凭空打分）
-评分依据必须是可观察的回复内容，禁止臆测
- 输出必须是 JSON格式，禁止任何推导过程外泄
-5维度必须独立评分，禁止相互干扰
-遇到 ground_truth 中标记为 P0 的样本，合规风控维度必须严格

## 输出 Schema
{
 "eval_id": "EVAL_0001",
 "scores": {
 "comprehensive_identification":0-3,
 "response_effectiveness":0-3,
 "tool_application":0-3,
 "copywriting_experience":0-3,
 "compliance_risk":0-3,
 "total_score":0-3,
 "rating": "S/A/B/C/D"
 },
 "badcase_level": "P0/P1/P2/无",
 "badcase_reasons": ["未识别盗刷", "未引导报案"],
 "error_category": ["FORMAT_ERROR", "WRONG_CHOICE", "MISSING_INFO"],
 "suggestion": "建议在 prompt 中加入明确的盗刷场景识别规则"
}

## LLM-as-Judge Rubric（文案体验维度）
5 分：专业有温度，主动提供帮助，情感共鸣，让人读完有"被理解"的感受
4 分：专业清晰，有明确服务意识，提供具体解决方案
3 分：专业清晰，有基本服务意识，但缺少主动关怀
2 分：语言清晰但缺乏温度，或有轻微措辞问题
1 分：语言混乱、冰冷、有明显语法/逻辑错误
```

###5.3迭代策略

评测 Agent本身也需要迭代（**评测系统 bug ≠ 被测 Agent bug**）：

| 版本 |阶段 |关键动作 |
|------|------|----------|
| **v1** |调试模式 | 带推导过程，方便排查逻辑错误 |
| **v2** |跑批模式 |切换为纯 JSON 输出，修复首批发现的评测逻辑 bug |
| **v3+** |调优模式 | 基于实际结果持续调优（指标定义、匹配方式、容错逻辑） |

###5.4评测系统常见 Bug 与修复

| 问题 |表现 |修复方式 |
|------|------|----------|
|匹配逻辑过严 |语义等价的判定原因被判错 | GT⊆AI 超集匹配 |
|硬编码规则误报 |排除列表不全导致误判 |改为动态语义比对 |
| Token截断 | 输出超长被评测平台截断 | 正则容错提取关键字段 |
| GT覆盖缺口 | 新增选项未在 GT 中体现 | 更新 GT标注 |
|评分漂移 | 不同时间评分不一致 |强化 Rubric锚定 +校准样本 |

###5.5 "评测 Agent调被测 Agent" 的踩坑经验

|坑 | 解法 |
|----|------|
|评测 Agent忘记调用工具 | 在 Constraints 中强调"必须先调用工具" |
|工具参数传递失败 | 在提示词中显式写明参数构造逻辑 |
|评测 Agent 重试耗尽 token | 添加"禁止重试"约束 |
| 输出截断 |减少推导过程，只输出最终 JSON |
| LLM-as-Judge偏差 | 对关键决策用人工抽检兜底 |

## 六、评测执行流程

###6.1 Harness 工程搭建式评测全流程

```
Step1：准备工作（5 分钟）
-准备好被测 Agent 的 prompt 文件
-确认被测 Agent 的接口信息和调用方式
- 在 Harness Agent 中打开项目目录

Step2：设计评测方案（10-30 分钟）
-告诉 Harness Agent：这是被测 Agent 的 prompt [粘贴/路径]，帮我设计评测方案
- Harness Agent 输出：完整评测方案 .md +边界用例建议

Step3：构建评测集（半天，含人工标注）
-告诉 Harness Agent：帮我从 XXX 接口拉取候选数据，格式化为评测集
- Harness Agent 输出：评测集 Excel（含 system.question 列）

Step4：编写评测 Agent提示词（1-2 小时）
-告诉 Harness Agent：基于评测方案，帮我写评测 Agent 的提示词
- Harness Agent 输出：评测 Agent提示词（含角色 +工具 +约束 + 工作流 + 输出 schema）

Step5：跑批 + 出报告（30 分钟）
- 上传评测集到平台 →等待跑批完成
-告诉 Harness Agent：跑批完了，结果在 XXX.xlsx，帮我出报告
- Harness Agent 输出：归因分析 +跨批次对比 + 可操作建议
```

###6.2评测节奏

|阶段 |频率 | 内容 |
|------|------|------|
| **日常迭代** | prompt改一行即跑 | 小批量快速验证（5-10 条边界 case） |
| **周度复盘** |每周一 | 全量评测集跑批 + 周报 |
| **月度评估** |每月1 号 | 全量 +行业对标 +评级调整 |
| **季度校准** | 每季度 |评测集增删 + Rubric调整 + Harness优化 |

###6.3评测平台调用

实际踩坑记录：

|坑 |现象 |解决方案 |
|----|------|----------|
| Token截断 | 输出超长被平台截断 |提示词中减少推导过程 + 正则容错提取 |
| API 超时 | 高并发下偶发超时 |平台层重试1 次 + 超时阈值30s |
| 并发限流 | 调用频次超限 |评测集分批（每批20 条）+间隔2s |
|成本失控 | 全量55 条 ×多次跑批 | 月度预算上限 + 单条成本监控 |

## 七、结果分析与报告

###7.1 自动分析的四项增值

|增值 | 说明 | 示例 |
|------|------|------|
| **自动识别 pattern** | 不只报数字，还归因 | "18 条误过滤中，12 条都是把某评分维度<60 当过滤条件" |
| **跨批次对比** | 和上一版结果对比，明确进步/退步 | "v3.1 vs v3.0：意图识别准确率 +5.2%，文案体验 -0.3 分" |
| **可操作建议** | 不只是"分数低"，而是"改哪里" | "建议在 prompt 第三段加入明确的过滤条件边界" |
| **Badcase聚类** | 按错误分类聚合，定位共性问题 | "FORMAT_ERROR12 条集中在长文本回复场景" |

###7.2评测报告模板

```
#智能客服评测报告 — v3.1（2026-06-XX）

## 执行情况
-评测集版本：v3.1_eval_set（55 条）
-跑批时间：2026-06-XX10:00 -10:35
-覆盖场景：P0紧急11 条 / P1常规22 条 / P2 长尾14 条 /边界6 条 / 反例2 条

##指标汇总
|指标 | 本次（v3.1） | 上次（v3.0） |变化 |
|------|--------------|--------------|------|
| 输出格式合规率 |100% |98% | +2% |
|字段完整率 |96% |92% | +4% |
|意图识别准确率 |94.5% |89.3% | +5.2% |
| 综合得分 |2.62 |2.45 | +0.17 |
|评级 | A 级 | A 级 |持平 |

## 问题分析
- P0 Badcase：0 条 ✅
- P1 Badcase：2 条（具体样本 ID +原因）
- P2 Badcase：5 条（聚类：3 条为文案体验不足，2 条为工具参数错误）

##改进建议
1. 在工具调度 prompt 中加入参数校验步骤（可消除2 条 P2）
2. 在文案生成 prompt 中加入共情表达模板（可提升文案体验均值）
3.评测集新增5 条长尾场景（理财 /外汇）

## 下次评测计划
- 时间：2026-06-15（周一）
-重点：验证改进建议落地效果
```

###7.3质量保障

- **覆盖性**：Harness Agent不会遗漏任何数据行（人工数50 行 Excel容易看漏）
- **一致性**：同样的评测标准，Harness Agent不会因为疲劳而评分漂移
- **溯源性**：每条评测结果都可追溯到 prompt 中的具体逻辑
- **可复现**：同一份评测 Agent提示词 +同一份评测集 = 结果可复现

## 八、错误分类体系（可扩展）

###8.1错误分类标准

|分类 | 说明 | 示例 |
|------|------|------|
| **FORMAT_ERROR** | 输出格式不符合规范 | JSON解析失败、字段缺失 |
| **WRONG_CHOICE** |分类/决策错误 |意图识别错误、工具选错 |
| **MISSING_INFO** |关键信息遗漏 | 未抽取关键槽位、未触发必要工具 |
| **HALLUCINATION** | 编造不存在的信息 |虚构费率、虚构政策 |
| **COMPLIANCE_VIOLATION** | 合规违规 | 未做风险提示、泄露隐私 |
| **TOOL_ERROR** |工具调用错误 | 参数错误、调用失败 |
| **COPY_QUALITY** | 文案质量问题 |语气生硬、缺乏共情 |

###8.2 Badcase 分级处理

|级别 |触发条件 | 处理方式 |
|------|----------|----------|
| **P0-Badcase** | 合规风控=0 或全面识别=0 |立即修复，24小时内上屏 |
| **P1-Badcase** | 综合<1.5 或 任一维度=0 | 次日修复，一周上屏 |
| **P2-Badcase** | 综合<2.0 或 文案体验<1 |排期修复，下个迭代 |
| **优化建议** | 综合≥2.0 |收集优化，纳入体验提升池 |

|阶段 |传统方式 | Harness协助 |加速比 |

|跑批执行 | 同（平台执行） | 同 |1x |
| 结果分析 +报告 |半天-1 天 |10-20 分钟 | ~5x |
| **单 Agent 全流程** | **~1.5 周** | **~1-2 天** | **~5x** |

###9.2关键收益

|收益 | 具体表现 |
|------|----------|
| **从周到天** | 单 Agent评测全流程从 ~1.5 周压缩到1-2 天 |
| **一人成军** |一个人 + Harness Agent 完成原来需要测试开发 + 数据标注 +分析师的工作 |
| **可持续迭代** |每次 prompt变更后的验证成本极低（改提示词 → 重跑 → 看报告） |
| **零部署** |评测逻辑是 Prompt 而非代码，无需 CI/CD，改完即生效 |
| **方法论沉淀** |指标框架 + Agent提示词模板 +错误分类体系，可迁移复用到任何 Agent |

###9.3适用场景

|场景 |适合度 |原因 |
|------|--------|------|
| **Prompt迭代验证** | ⭐⭐⭐⭐⭐ |改 prompt →跑批 → 看报告，闭环最快 |
| **多 Agent横向对比** | ⭐⭐⭐⭐⭐ |统一指标框架 +相同评测流程 |
| **新 Agent 上线前验收** | ⭐⭐⭐⭐ | 系统性覆盖，不依赖人工抽检 |

###9.4局限与建议

|局限 |建议 |
|------|------|
| LLM-as-Judge本身有偏差 | 对关键决策用人工抽检兜底 |
|评测集规模受限（人工 GT） | 小而精优于大而糙，20-55 条覆盖边界即可 |
|依赖评测平台稳定性 | token截断、API 超时需做容错 |
|首次搭建有学习成本 |第二个 Agent 起复用率很高 |
