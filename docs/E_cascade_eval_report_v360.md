# Cascade 三级回退实测报告 — D_eval_set_v3.2

**版本**: v3.6.0-Cascade-L2 on D-v3.2
**评测集**: `D_eval_set_v3.2.json` (1500 条黄金评测集, 21/21 P0, 86 意图覆盖)
**模型**: L1 规则 (IntentRecognizer) + L2 BERT (`models/M3-bert-base-chinese`) + L3 LLM 兜底
**门限**: 置信度 ≥ 0.85 直返, 否则下一级
**实测时间**: 24 分钟 (1500 条单进程 CPU)
**测试日期**: 2026-06-21

## 7. 关键问题诊断

### 7.1 label 空间不一致（工程债）

| 维度 | D v3.2 (新) | IntentRecognizer (旧) |
|---|---|---|
| label 总数 | 86 | 99 |
| 前缀集合 | biz/consult/info/mkt/safety/security/sys (7) | biz/cons/info/sales/sec/sys (6) |
| 命名差异 | `consult_credit_card_limit` | `cons_cred_limit` (缩写) |
| 缺失组 | safety (反诈) 完整 | **完全无对应** |
| 缺失意图 | D 新增细类 (P0 变体) | **无训练样本** |

**根因**: A_standard v3.2 重构成 7 个业务组 + 86 个细类, 但 IntentRecognizer 还停留在 v3.5.x 的 99 个旧 label。

### 7.2 L2 BERT 几乎没起作用 (3.2%)

- M3-bert-base-chinese 在 v3.5.x label 空间训练
- D v3.2 新增了 86 个 label, 但模型不认
- 后果: L1 兜不住的 36.7% L3 兜底, L2 命中率只有 3.2%

### 7.3 P0 红线召回 26% (银行业不可接受)

- 827 条 P0 中只有 215 条被分到正确业务组
- safety 组 (257 条反诈骗/反洗钱) **100% 走 L3 LLM 兜底**——意味着所有反诈问题都丢给外部 LLM, **数据外泄 + 合规风险**
- 必须通过代码层 (非 LLM) 拦截 P0

## 8. 提升方案（按优先级）

### P0 紧急 (本周)
1. **补充 IntentRecognizer safety 组意图** —— 加 `safety_fraud_sms / safety_fraud_call / safety_fraud_transfer` 等 5-8 个 P0 规则
   - 预期: P0 召回 26% → 50%+
2. **P0 强制转人工** (C_eval_system §三 红绿灯) —— 检测到 P0 关键词直接 `route_human`, 不走 L3
   - 预期: P0 召回 50% → 76%

### P1 中期 (2 周内)
3. **统一 label 空间** —— A_standard v3.2 的 86 label 作为唯一权威, 废弃 IR 老 label
4. **M3-bert 用 D v3.2 重训** —— label 对齐后细类匹配应该从 2.53% → 60%+

### P2 长期 (1 月内)
5. **L2 命中优化** —— 期望 L2 从 3.2% → 15-20%, L3 兜底从 36.7% → 20% 以内
6. **Cascade 路由版本对比** —— v3.6.0 vs v3.7.0 在同一评测集上的统计验证
