### 2.1 关于"业务等价口径 92.55% vs v3.6.3 96.51%"的诚实修正

v3.6.3 报告里"业务等价口径 96.51%"包含 `safety_card_loss ↔ security_fraud_report` 双意图互抢的 47 条 v3.6.3 design behavior miss。v3.6.4 这次有两个变化：

1. **biz_transfer_large 全新 intent** 让 22 条 P0 query 从 `security_aml_large_transfer` group 转入 `biz_transfer_large` group
   - 业务上仍然是 P0 转人工（业务等价口径应该算），但**评测口径的 group_match 从 security_aml 变成 biz**
   - v3.6.4 的 P0 业务等价口径不再把 sec_fraud_report 这种 IR 老 label 算入（因为 v3.6.4 已经让 D v3.2 label 直接命中）

2. **口语化转人工 patterns 让 sys_service_route_human 多命中 31 条**
   - 这些 query 在 v3.6.3 走 sys_invalid / sys_greeting / sys_thanks（不触发 should_transfer）
   - v3.6.4 让它们正确命中 sys_service_route_human（should_transfer=True）
   - 业务口径 + 评测口径**同时受益**

**PM 视角的诚实口径**：
- **评测口径 91.92%** — 真实业务场景下 P0 query 至少分到了正确业务组（PM 决策 KPI）

面试话术：诚实说明"三个口径各有意义——评测口径看业务组归属，业务口径看 should_transfer 触发，业务等价口径含设计等价。我选评测口径作为 KPI，因为它最严格、最不易被'打补丁作弊'。"

### 8.2 面试高频问题（新增 v3.6.4）

**Q1. v3.6.4 三个补丁怎么挑出来的？为什么按这个 ROI 排序？**

按 v3.6.3 报告 §5 的 127 条 P0 业务 miss 根因分析，三个 ROI 最高的补丁：

1. **biz_transfer_large 全新 intent（0% → 90.9%）**：v3.6.4 P0 4-groups 报告发现 22 条 biz_transfer_large 命中 security_aml_large_transfer（业务等价但评测口径错）。新增独立 intent 让评测口径归位，1 个 patch 修 20 条 miss
2. **security_fraud_report 反诈口语化（69.1% → 86.4%）**：v3.6.3 漏判 11 条口语化变体（"假冒公安/骗验证码/电信诈骗"），加 52 个 patterns
3. **sys_service_route_human 口语化（业务口径 80.9% → 84.7%）**：v3.6.3 漏判 31 条口语化（"找真人/95555/客服解决不了"），加 58 个 patterns

为什么 P2 排第二但 ROI 最高？——全新 intent 是结构性问题（IR label 缺失），一次投入永久受益，比 patterns 扩展的边际收益大。

**Q2. v3.6.3 报告说 P0 业务等价口径 96.51%，v3.6.4 反而降了到 92.55%，是不是改坏了？**

**不是改坏，是诚实修正口径含义**。v3.6.3 的 96.51% 包含 sec_fraud_report 这种 IR 老 label 的"业务等价命中"（47 条设计等价）。v3.6.4 让 D v3.2 label (security_fraud_report / biz_transfer_large) **直接命中**，不再需要 IR 降级映射，所以业务等价口径这个口径的"基数"变了。

**两个口径同时看才能反映真实业务**：
- v3.6.3: 业务口径 87.80% / 业务等价 96.51% (设计等价 P0 多)
- v3.6.4: 业务口径 90.81% / 业务等价 92.55% (评测口径更严，但业务口径+3.01pp 才是真 KPI)

面试官问起来能答出"我知道 v3.6.4 业务等价口径降是因为口径基数变了，真实业务口径提升了 3pp"——这种口径的精确理解比"我们 P0 召回 100%"更有说服力。

**Q3. biz_transfer_large 全新 intent 怎么保证不会和 security_aml_large_transfer 抢匹配（"被骗了怎么办"这种 query）？**

v3.6.4 dry-run 时确实发现"被骗了怎么办"被 biz_transfer_large 的 "怎么办" pattern 抢匹配。**修复方案**：在 security_fraud_report 里加 5 条精确复合 query（"被骗了怎么办"/"被诈骗了怎么办"/"被人骗了怎么办"等），让反诈意图抢先。

**PM 视角的优先级设计**：反诈 > 卡片资产 > 大额业务（v3.6.3 报告 §4 双意图设计已说明）。**当反诈 query 撞到通用词（"怎么办"/"需要什么"），让反诈抢匹配**——这是业务红线，业务上涉及诈骗必须转反诈中心而非业务办理。

**Q4. 为什么 v3.6.4 不继续优化到 95%+？**

剩余 51 条 miss（7.08%）分布如下：

- 25 条 sys_service_route_human 极端口语化（"我那个…钱的事…有点…那个"）——需要 LLM 兜底理解，词典无法解决
- 5 条 prompt injection（"绕过密码"/"忽略指令"）——需要 prompt 防御层 + LLM 兜底
- 14 条 safety ↔ fraud 双意图设计行为（业务等价 P0）
- 5 条 sys_service_complaint 单字"投诉"——需要 word-boundary regex 重构 L1 规则
- 2 条 biz_transfer_large 短 query

**v3.6.5 候选预期**：补 sys_service_route_human 口语化 (16 条) + biz_transfer_large 短 query (2 条) + sys_service_complaint 短词 (5 条) = **预期 P0 召回 → 94-96%**。

剩下 4-6pp 是 LLM 兜底和 prompt 防御层的范围，**需要更多工程投入（1-2 周），而非简单的 patterns 扩展**。
