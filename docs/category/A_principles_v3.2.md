# 智能客服三级类目分类原则 v3.2

> 文档目的：定义 **5 条分类原则** + 边界规则，作为 NLU 标注员和对话设计师判断 query 归属的判定依据。
>
> 适用版本：v3.2

---

## 原则 1：合规 / 消费者权益保护 > 高频易标准化 > 低频难个性化

### 优先级量化

| 优先级 | 数量 | 含义 | 判定标准 |
|---|---|---|---|
| P0 | 21 | 必答 / 必转 / 必拦截 | 合规强制 / 监管报送 / 高风险反洗钱 |
| P1 | 38 | 高频标准回答 | 日均 ≥ 100 次 / 答案标准化程度高 |
| P2 | 20 | 低频个性化 | 日均 < 100 次 / 答案需结合客户情况 |
| P3 | 3 | 闲聊无效 | 闲聊/乱码/再见 |

### 判定流程

```
query 进入 → 是否触发合规/监管/反洗钱？
            ├─ 是 → P0（强转人工 / 模板回复 / 拦截）
            └─ 否 → 是否高频？
                     ├─ 是（≥100 次/日）→ P1
                     ├─ 否（<100 次/日）→ P2
                     └─ 闲聊/乱码 → P3
```

### 原则 1 体现

- 21 个 P0 中 **15 个是合规/监管/反洗钱**（占比 71%）
- SECURITY 域 100% P0（反诈/反洗钱/适当性/承诺收益都是监管强制）
- 投诉 P0（监管要求 100% 人工受理）

## 原则 2：临时 vs 常驻判断准则

| 维度 | 临时（→ MARKETING）| 常驻（→ CONSULT）|
|---|---|---|
| **时效** | 有起止日期/报名期/倒计时 | 长期有效（无截止日期）|
| **频次** | 周/月/季度更新 | 半年以上不变 |
| **规则** | 活动规则（抽奖/满减/限时）| 产品规则（利率/费率/条款）|
| **来源** | 运营临时配置 | 银行产品部正式发文 |
| **典型 query** | "周三 5 折怎么用" / "M+ 本月有什么福利" | "美元汇率今天多少" / "信用卡年费多少" |

### 原则 2 体现

- **MARKETING 11 个三级全部是临时活动**（每周/每月/季度/活动期）
- **CONSULT 22 个三级全部是常驻规则**（利率/费率/条款/规则）
- **关键边界**：
  - 加息券活动 → `mkt_pay_coupon`（临时）
  - 加息规则说明 → `consult_wealth_deposit`（常驻）

## 原则 3：SECURITY vs SAFETY 拆分逻辑

| 维度 | SECURITY（合规红线）| SAFETY（账户安全）|
|---|---|---|
| **对象** | 对监管的 | 对用户的 |
| **触发源** | 监管报送/合规要求 | 客户应急需求 |
| **动作** | 100% P0 强转人工 + 监管报送 | P1 正常办理 + P0 仅挂失 |
| **典型场景** | 反诈/反洗钱/适当性/承诺收益 | 冻结/挂失/密码/设备 |

### 原则 3 体现

- **SECURITY 7 个三级**（反诈 2 + 反洗钱 2 + 适当性 2 + 承诺收益 1）**100% P0**
- **SAFETY 5 个三级**（卡状态 2 + 密码 2 + 设备 1）**仅 1 个 P0**（挂失）
- 一个对监管、一个对用户，**逻辑边界清晰**

### 联动机制

| SAFETY 触发 | SECURITY 联动 | 业务含义 |
|---|---|---|
| `safety_card_loss`（挂失）| `security_fraud_recognize` | 🔥 挂失必联动反诈核实 |
| `safety_password_forget`（疑似盗用）| `security_fraud_recognize` | 🔥 密码疑似被盗→反诈 |

## 原则 4：BIZ 主动办 vs SAFETY 异常态（密码分流规则）

> **为什么需要这条原则**？
> "改密码"在客户语境下有 3 种含义：（1）主动改密码；（2）密码忘了要重置；（3）密码异常（被锁/被盗）。**3 种属于不同三级**，分流错了会出合规问题。

### 密码分流决策表

| 客户表达 | 归属 | 三级 | 判断依据 |
|---|---|---|---|
| "我要改密码" | BIZ | `biz_password_reset` | 主动办理 |
| "怎么修改银行卡密码" | BIZ | `biz_password_reset` | 主动办理 |
| "我登录密码忘了怎么重置" | BIZ | `biz_password_reset` | 目标 = 重置（操作类，虽然"忘了"但目标是重置）|
| "我密码输错被锁了" | SAFETY | `safety_password_locked` | 异常态 |
| "我密码好像被人改过" | SAFETY + SECURITY | `safety_password_forget` + **双触发 `security_fraud_recognize`** | 疑似盗用 |

### 决策流程

```
query 涉及密码
  ├─ 主动办（"我要改"/"怎么改"/"忘了怎么重置"）→ BIZ.biz_password_reset
  └─ 异常态（"被锁"/"被盗"/"被人改过"）→ SAFETY
         ├─ "被锁" → safety_password_locked (P1)
         └─ "被盗/被人改过" → safety_password_forget + 双触发 SECURITY.fraud_recognize (P0)
```

### 同类扩展：手机解绑分流（配合原则 5）

| 客户表达 | 归属 | 判断依据 |
|---|---|---|
| "我想换绑手机" | `safety_device_unbind` | 主动办（操作类）|
| "我手机丢了" | `safety_device_unbind` + 双触发 `security_fraud_recognize` | 设备异常→盗用风险 |

## 原则 5：多意图 query 的 disambiguation 规则（v3.2 新增）🔥

> **为什么需要这条原则**？
> 客户的真实 query 经常是模糊的——"跨行转账怎么操作"既可能是咨询（怎么操作 = 想知道流程），也可能是想办（想办但不知道怎么办）。**一刀切归到 CONSULT 或 BIZ 都会出错**。

### 业界方法对比

| 方法 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| **A. 强制单意图分类** | 简单 | 答非所问 | 历史训练数据足够、意图清晰 |
| **B. 多意图并列回答** | 全 | 长、慢、用户抓不住重点 | 极少数确实两个都要的场景 |
| **C. 主意图回答 + 澄清追问 + 动作入口** ✅ | 准、转化高 | 多一次交互 | **历史数据不足、意图模糊、跨域多意图** |
| **D. 概率分布 + 默认高置信意图** | 自动 | 黑盒 | 数据量大、阈值可调 |

**我们选 C**——**回答 + 澄清追问 + 动作入口**（用户原话："回答之后多问一句类似'您是想要办理跨行转账业务吗？'并给出跨行转账的业务入口"）

### Disambiguation 模式（3 步法）

```
Step 1：识别主意图（按 BERT 置信度排序，取 Top1）
Step 2：先答 Top1 的内容（不让用户白等）
Step 3：判断 Top1 置信度
         ├─ 高（≥0.85）→ 单意图直接答
         └─ 中/低（<0.85）→ 追加 disambiguation：
              "您是想<动作 A>，还是<动作 B>？"
              + 给出对应入口按钮（"是 → 点这里办" / "否 → 继续问"）
```

### 关键规则

| 规则 | 说明 |
|---|---|
| **必须先答主意图** | 不让用户白等（不是只问不答）|
| **追问必须给入口** | "是 → 点这里"按钮 / "否 → 继续问"，**不能只问不引导** |
| **不重复问相同问题** | 一次 disambiguation 失败，直接转人工 |
| **置信度阈值** | 0.85（与 BERT L2 门控一致）|
| **适用范围** | 跨域多意图 / 信息不足 / 历史数据无该 query |

### 8 个高风险多意图 query 全表复查

| # | 客户 query | 主意图（Top1）| 次意图（Top2）| Disambiguation 模式 |
|---|---|---|---|---|
| 1 | "跨行转账怎么操作" | consult（咨询流程）| biz_transfer_cross_bank（办理）| 答完问"是想要现在办跨行转账吗？" + `/api/payment/transfer-cross` 入口 |
| 2 | "怎么改密码" | biz_password_reset（改密码）| safety_password_locked（密码异常）| 答完问"是忘了密码要重置，还是密码异常？" + 3 个分支入口 |
| 3 | "怎么解绑手机" | safety_device_unbind（解绑）| safety_device_unbind + 双触发（疑似盗用）| 答完问"是要换绑手机，还是怀疑被盗？" + 2 个入口 |
| 4 | "我卡被冻结了" | safety_card_freeze（状态）| biz_card_activate（解冻）| 答完问"是查询冻结原因，还是要解冻？" + 查询/解冻入口 |
| 5 | "贷款进度" | biz_loan_apply（查进度）| sys_service_complaint（催办/投诉）| 答完问"是要催办，还是查询？" + 催办/查询入口 |
| 6 | "我密码好像被人改过" | safety_password_forget（异常）| 🔥 **双触发 SECURITY.fraud_recognize** | **必须** P0 强转人工 + 反诈中心 |
| 7 | "信用卡额度怎么提" | consult_credit_card_limit（条件）| biz_credit_card_apply（申请）| 答完问"是查询提额条件，还是要现在申请？" + 申请入口 |
| 8 | "我要转账" | biz_transfer（转账）| security_aml_large_transfer（大额反洗钱）| 自动识别金额，>5 万自动触发 SECURITY |

### 兜底：`sys_other_unclear`

> 当 query 命中 0 意图或多种意图概率都 < 0.5 时：
> - 不直接答非所问
> - 走 disambiguation 兜底：列出 2-3 个最可能意图 + 入口
> - 一次兜底失败 → 转 `sys_service_route_human` 人工

### 原则 5 体现

- 🔥 全表复查识别 **8 个高风险多意图 query**
- 每个 query 都配置了 **disambiguation 模式**（答 + 问 + 入口）
- 兜底走 `sys_other_unclear` → `sys_service_route_human` 人工

## 5 条原则的优先级与冲突解决

### 原则冲突时如何处理

| 冲突场景 | 处理方式 | 依据 |
|---|---|---|
| 原则 1（P0 合规）vs 原则 2（临时 vs 常驻）| **P0 优先** | 合规 > 业务分类 |
| 原则 3（SECURITY）vs 原则 4（密码分流）| **联动不冲突** | 异常态密码自动双触发 SECURITY |
| 原则 2（活动临时）vs 原则 5（多意图）| **先按原则 2 分类，再加 disambiguation** | 业务优先，UX 兜底 |
| 原则 4（密码分流）vs 原则 5（多意图）| **先按原则 4 分流，再加 disambiguation** | 合规优先，UX 兜底 |

### 一句话总结

> **合规第一**（原则 1）→ **业务清晰**（原则 2/3/4）→ **UX 兜底**（原则 5）
